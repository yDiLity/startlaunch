Инструкция для Code Agent: Система Memory Bank
Ты действуешь как инженер, который не обладает долгосрочной памятью. 
Твой единственный устойчивый контекст — это директория /memory_bank. 
Твоя задача — поддерживать документацию в актуальном состоянии, синхронизируя её с кодом.

1. Фундаментальные правила (Policy)
 Источник архитектурной правды: Файл docs/README.md — единственный источник информации о высокоуровневой архитектуре.
• Канонический источник процента выполнения проекта: `memory_bank/projectbrief.md`.
• В `memory_bank/projectbrief.md` обязателен отдельный раздел `## Project Deliverables`.
• Агент обязан автоматически создать раздел `## Project Deliverables`, если он отсутствует, как только задача затрагивает оценку прогресса проекта, AI-анализ проекта, planning проекта или синхронизацию `memory_bank`.
• Агент обязан сам формировать и поддерживать `## Project Deliverables` на основе `docs/README.md`, связанных ТЗ, подтвержденных product requirements и фактического состояния кода.
• `## Project Deliverables` — это не журнал сессии, а канонический список обязательных результатов проекта, по которому считается процент выполнения.
• Каждый deliverable в `## Project Deliverables` обязан иметь:
  - стабильный ID;
  - краткое название результата;
  - статус `pending`, `in_progress`, `completed` или `blocked`;
  - вес в процентах.
• Сумма весов всех deliverables в `## Project Deliverables` должна составлять `100`.
• Процент выполнения проекта считается только по `## Project Deliverables` из `memory_bank/projectbrief.md`.
• `activeContext.md`, `progress.md` и `Changelog` используются как операционный контекст и история изменений, но не как источник процента выполнения проекта.
• Каждый новый пункт ТЗ, обязательный этап реализации или подтвержденный scope change должен быть добавлен агентом в `## Project Deliverables` до начала реализации или одновременно с ней.
• После каждой задачи, которая влияет на готовность проекта, агент обязан обновить статусы deliverables в `memory_bank/projectbrief.md`.
• Если deliverable разбивается на подэтапы, в процент проекта идет только итоговый статус самого deliverable, а не записи из `Changelog` или промежуточные заметки в `activeContext.md`.
• Если deliverables отсутствуют, не покрывают текущий scope проекта или не синхронизированы с кодом, агент обязан сначала восстановить и обновить `## Project Deliverables`, и только потом рассчитывать процент выполнения.
• Пока `## Project Deliverables` не приведен в актуальное состояние, процент выполнения проекта должен считаться как `нет достоверных данных`, а не вычисляться по эвристикам.

2. Любые изменения в архитектуре, маршрутах или модулях должны быть отражены здесь.
 Память проекта (Memory Bank):
    ◦ Перед началом любой работы перечитай содержимое /memory_bank, особенно activeContext.md и progress.md.
    ◦ /memory_bank — это твоя операционная система. Без актуализации этих файлов ты теряешь контекст.

3. Контроль изменений:
    ◦ Раздел «Контроль изменений» в файле progress.md всегда должен содержать хеш или дату последнего проверенного коммита (last_checked_commit).
    ◦ При каждом сеансе сравнивай текущее состояние с последним проверенным: git log <last_checked_commit>...
    ◦ Если твои действия затрагивают документацию, обнови хеш последнего коммита.

4. Структура файловой системы Memory Bank
Ты обязан поддерживать следующую иерархию файлов в папке /memory_bank:
• projectbrief.md — Фундаментальный документ: цели, рамки проекта.
• productContext.md — Продуктовый контекст: зачем нужен проект, проблемы пользователей.
• activeContext.md — Текущий фокус: над чем работаем прямо сейчас, активные решения и приоритеты.
• systemPatterns.md — Технические решения: архитектура, паттерны, связи подсистем.
• techContext.md — Стек технологий: окружение, библиотеки, ограничения, CI/CD.
• progress.md — Статус: что готово, известные проблемы (Known Issues), история изменений и last_checked_commit.
Дополнительные разделы (по необходимости):
• modules/* — Детальные знания по конкретным модулям (API, границы).
• ui_extension/* — Детали UI: страницы, компоненты, UX-паттерны.
• other/* — Специфические заметки (например, legacy-код, настройки php_interface).

5. Рабочие режимы (Workflows)

Режим А: Планирование задачи (До написания кода)
Прежде чем писать код для крупных, архитектурных, многосоставных или рискованных задач, ты должен утвердить план.
Для повседневых правок UI, точечных фиксов, локальных рефакторингов без неочевидных последствий и других небольших изменений отдельное согласование плана не требуется: агент должен сразу выполнять задачу, если запрос пользователя однозначен и scope понятен.
1. Анализ: Прочитай productContext.md и activeContext.md. Найди релевантные разделы в модульной документации.
2. Формирование плана: Составь пошаговый алгоритм реализации.
    ◦ Укажи каждый шаг и ожидаемый результат.
    ◦ Перечисли файлы, которые будут затронуты.
    ◦ Опиши риски или неопределенности.
3. Остановка: Не приступай к выполнению, пока план не будет одобрен пользователем, только если задача относится к крупным, архитектурным, многосоставным или рискованным изменениям.

Режим Б: Выполнение и Документирование
После утверждения плана для крупных задач или сразу после анализа для повседневных правок:
1. Синхронизация: Обнови activeContext.md, указав, что задача взята в работу.
2. Реализация: Выполняй работу последовательно, шаг за шагом.
3. Обновление памяти:
    ◦ Если изменилась структура CI/CD (например, .gitlab-ci.yml), немедленно обнови techContext.md и systemPatterns.md.
    ◦ Если добавлен новый функционал, создай ТЗ или обнови описание в docs/README.md со ссылкой на детали.
    ◦ Если найдены "опасные" места или legacy, зафиксируй их в other/ и добавь пометку в progress.md.
4. Фиксация: После завершения этапа обнови progress.md (раздел Changelog) и запиши новый last_checked_commit.

Режим В: Инициализация (Если Memory Bank пуст)
Если папки /memory_bank нет или она пуста:
1. Создай папку и обязательные файлы-заглушки (projectbrief.md... progress.md).
2. Проведи первичный ресёрч репозитория (ограниченный контекст): составь карту основных директорий и точек входа.
3. Запиши результаты карты в techContext.md и systemPatterns.md. Не углубляйся в детали кода на этом этапе.
4. Специфические инструкции по обновлению
• UI и Страницы: Для каждой публичной страницы создавай файл в ui_extension/pages/, описывая назначение, используемые компоненты и потоки данных.
• Компоненты и Шаблоны: Компоненты описывай в ui_extension/components/. 
Если невозможно точно определить активный шаблон (настройка в админке), зафиксируй гипотезу в progress.md как «известную неопределенность».
• Технические задания (ТЗ): Новые фичи требуют отдельного ТЗ в документации. 
В основных файлах (docs/README.md) должны быть только ссылки на эти ТЗ, чтобы не перегружать контекст.
