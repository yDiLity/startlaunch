# Документ требований

## Введение

AutoLaunch — это кроссплатформенное десктопное приложение, которое автоматически анализирует, настраивает и запускает проекты с GitHub. Пользователь просто вставляет ссылку на репозиторий, а приложение определяет тип проекта, устанавливает зависимости, настраивает окружение и запускает проект, предоставляя результат в браузере или как нативное приложение.

## Глоссарий

- **AutoLaunch**: Основное десктопное приложение
- **Репозиторий**: Проект на GitHub, который пользователь хочет запустить
- **Песочница**: Изолированное окружение для безопасного запуска кода (Docker контейнер)
- **Прямой режим**: Запуск в виртуальном окружении языка без полной изоляции
- **Снимок проекта**: Сохраненное состояние настроенного проекта для быстрого перезапуска
- **Стек технологий**: Определенный набор технологий проекта (Python, Node.js, Rust и т.д.)

## Требования

### Требование 1

**Пользовательская история:** Как пользователь, я хочу ввести URL GitHub репозитория, чтобы система могла его проанализировать и подготовить к запуску.

#### Критерии приемки

1. КОГДА пользователь вводит валидный GitHub URL ТОГДА система ДОЛЖНА извлечь владельца и имя репозитория
2. КОГДА пользователь вводит невалидный URL ТОГДА система ДОЛЖНА отобразить понятное сообщение об ошибке
3. КОГДА пользователь вводит URL в формате owner/repo ТОГДА система ДОЛЖНА автоматически преобразовать его в полный GitHub URL
4. КОГДА система получает URL ТОГДА система ДОЛЖНА клонировать репозиторий в локальную временную директорию
5. КОГДА клонирование завершается ТОГДА система ДОЛЖНА сохранить информацию о проекте в локальной базе данных

### Требование 2

**Пользовательская история:** Как пользователь, я хочу, чтобы система автоматически определила тип проекта, чтобы не настраивать окружение вручную.

#### Критерии приемки

1. КОГДА система анализирует структуру файлов ТОГДА система ДОЛЖНА определить основной стек технологий проекта
2. КОГДА в корне проекта присутствует package.json ТОГДА система ДОЛЖНА определить проект как Node.js
3. КОГДА в корне проекта присутствует requirements.txt или pyproject.toml ТОГДА система ДОЛЖНА определить проект как Python
4. КОГДА в корне проекта присутствует Cargo.toml ТОГДА система ДОЛЖНА определить проект как Rust
5. КОГДА в корне проекта присутствует Dockerfile ТОГДА система ДОЛЖНА определить проект как Docker-based

### Требование 3

**Пользовательская история:** Как пользователь, я хочу, чтобы система автоматически определила команду запуска, чтобы не изучать документацию проекта.

#### Критерии приемки

1. КОГДА система анализирует package.json ТОГДА система ДОЛЖНА извлечь команду запуска из поля scripts.start
2. КОГДА в проекте присутствует main.py ТОГДА система ДОЛЖНА предложить команду "python main.py"
3. КОГДА в проекте присутствует index.js без package.json ТОГДА система ДОЛЖНА предложить команду "node index.js"
4. КОГДА система не может определить команду запуска ТОГДА система ДОЛЖНА проанализировать README.md для поиска инструкций
5. КОГДА команда запуска не найдена ТОГДА система ДОЛЖНА запросить у пользователя ввод команды вручную

### Требование 4

**Пользовательская история:** Как пользователь, я хочу запускать проекты в безопасном изолированном окружении, чтобы защитить свою систему от потенциально вредоносного кода.

#### Критерии приемки

1. КОГДА пользователь запускает неизвестный проект ТОГДА система ДОЛЖНА по умолчанию использовать режим песочницы
2. КОГДА система создает Docker контейнер ТОГДА система ДОЛЖНА ограничить права контейнера (no-root, read-only FS)
3. КОГДА система обнаруживает потенциально опасные команды ТОГДА система ДОЛЖНА показать предупреждение пользователю
4. КОГДА пользователь подтверждает доверие к проекту ТОГДА система ДОЛЖНА сохранить этот статус для будущих запусков
5. КОГДА Docker недоступен ТОГДА система ДОЛЖНА использовать альтернативные механизмы изоляции

### Требование 5

**Пользовательская история:** Как пользователь, я хочу видеть процесс установки и запуска, чтобы понимать что происходит и диагностировать проблемы.

#### Критерии приемки

1. КОГДА система выполняет операции ТОГДА система ДОЛЖНА отображать текущий этап в прогресс-баре
2. КОГДА система устанавливает зависимости ТОГДА система ДОЛЖНА показывать логи установки в реальном времени
3. КОГДА происходит ошибка ТОГДА система ДОЛЖНА отобразить понятное сообщение об ошибке с предложениями решения
4. КОГДА пользователь нажимает на детали ТОГДА система ДОЛЖНА показать полные технические логи
5. КОГДА операция завершается успешно ТОГДА система ДОЛЖНА показать статус "Готово" с зеленой галочкой

### Требование 6

**Пользовательская история:** Как пользователь, я хочу управлять жизненным циклом запущенного проекта, чтобы контролировать его выполнение.

#### Критерии приемки

1. КОГДА проект запущен ТОГДА система ДОЛЖНА предоставить кнопки "Остановить", "Перезапустить", "Приостановить"
2. КОГДА пользователь нажимает "Остановить" ТОГДА система ДОЛЖНА корректно завершить все процессы проекта
3. КОГДА пользователь нажимает "Перезапустить" ТОГДА система ДОЛЖНА остановить и заново запустить проект
4. КОГДА проект остановлен ТОГДА система ДОЛЖНА автоматически очистить временные ресурсы
5. КОГДА пользователь закрывает приложение ТОГДА система ДОЛЖНА предложить остановить все запущенные проекты

### Требование 7

**Пользовательская история:** Как пользователь, я хочу сохранять настроенные проекты, чтобы быстро запускать их повторно без переустановки зависимостей.

#### Критерии приемки

1. КОГДА проект успешно запущен ТОГДА система ДОЛЖНА предложить сохранить снимок проекта
2. КОГДА пользователь сохраняет снимок ТОГДА система ДОЛЖНА сохранить все зависимости и конфигурацию в локальную директорию
3. КОГДА пользователь открывает сохраненный снимок ТОГДА система ДОЛЖНА запустить проект менее чем за 10 секунд
4. КОГДА система создает снимок ТОГДА система ДОЛЖНА сохранить метаданные (команды, порты, переменные окружения)
5. КОГДА пользователь удаляет снимок ТОГДА система ДОЛЖНА полностью очистить все связанные файлы

### Требование 8

**Пользовательская история:** Как пользователь, я хочу видеть историю запущенных проектов, чтобы быстро находить и перезапускать ранее использованные репозитории.

#### Критерии приемки

1. КОГДА пользователь запускает проект ТОГДА система ДОЛЖНА добавить его в историю
2. КОГДА пользователь открывает менеджер проектов ТОГДА система ДОЛЖНА показать список всех проектов с датами последнего запуска
3. КОГДА пользователь ищет проект по имени ТОГДА система ДОЛЖНА фильтровать список проектов
4. КОГДА пользователь добавляет теги к проекту ТОГДА система ДОЛЖНА сохранить эти теги для организации
5. КОГДА пользователь нажимает на проект в истории ТОГДА система ДОЛЖНА предложить быстрый перезапуск

### Требование 9

**Пользовательская история:** Как пользователь, я хочу настраивать поведение приложения, чтобы адаптировать его под свои предпочтения и рабочий процесс.

#### Критерии приемки

1. КОГДА пользователь открывает настройки ТОГДА система ДОЛЖНА показать опции режима изоляции по умолчанию
2. КОГДА пользователь изменяет путь сохранения снимков ТОГДА система ДОЛЖНА использовать новый путь для всех последующих операций
3. КОГДА пользователь выбирает темную тему ТОГДА система ДОЛЖНА применить темную цветовую схему ко всему интерфейсу
4. КОГДА пользователь включает автоочистку ТОГДА система ДОЛЖНА автоматически удалять временные файлы после остановки проектов
5. КОГДА пользователь сохраняет настройки ТОГДА система ДОЛЖНА сохранить их в конфигурационный файл

### Требование 10

**Пользовательская история:** Как пользователь, я хочу видеть запущенные веб-приложения в браузере, чтобы взаимодействовать с ними как с обычными веб-сайтами.

#### Критерии приемки

1. КОГДА веб-приложение запускается ТОГДА система ДОЛЖНА автоматически определить порт из логов или конфигурации
2. КОГДА порт определен ТОГДА система ДОЛЖНА автоматически открыть браузер на localhost:port
3. КОГДА приложение использует нестандартный порт ТОГДА система ДОЛЖНА корректно его обнаружить
4. КОГДА приложение не запускается на ожидаемом порту ТОГДА система ДОЛЖНА показать сообщение с инструкциями
5. КОГДА пользователь закрывает браузер ТОГДА система ДОЛЖНА продолжать работу приложения в фоне
