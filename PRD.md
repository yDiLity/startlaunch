1. Обзор приложения и цели

1.1. Проблема

Разработчики, исследователи и энтузиасты часто сталкиваются с трудностями при запуске проектов с GitHub. Плохая документация, отсутствующие зависимости, сложные шаги настройки и различия в окружениях приводят к потере времени и разочарованию. Текущие решения (Docker, ручная настройка) требуют технических знаний и не предоставляют единого простого интерфейса.

1.2. Решение

AutoLaunch — это кроссплатформенное десктопное приложение, которое автоматически анализирует, настраивает и запускает проекты с GitHub. Пользователь просто вставляет ссылку на репозиторий, а приложение определяет тип проекта, устанавливает зависимости, настраивает окружение и запускает проект, предоставляя результат в браузере или как нативное приложение.

1.3. Цели продукта

· Основная: Упростить и ускорить запуск любого проекта с GitHub до одного клика.
· Пользовательская: Сделать технологию доступной для пользователей любого уровня подготовки (от новичков до экспертов).
· Техническая: Обеспечить безопасный и изолированный запуск произвольного кода.
· Бизнес-цель (потенциальная): Стать стандартным инструментом в workflow разработчиков при тестировании и использовании open-source проектов.

1.4. Ключевые метрики успеха

· Время от вставки ссылки до запуска проекта < 5 минут для 90% типовых проектов.
· Успешный запуск > 80% популярных репозиториев из Top-1000 GitHub.
· Показатель возврата пользователей (Retention) > 40% за 30 дней.
· Положительные отзывы о простоте использования и безопасности.

2. Целевая аудитория

2.1. Основные сегменты

1. Начинающие разработчики и студенты
   · Потребности: Легкий способ запускать учебные и примеры проектов без глубоких знаний о DevOps.
   · Боли: Страх командной строки, сложности с настройкой сред, непонимание документации.
2. Опытные разработчики и инженеры
   · Потребности: Быстро протестировать библиотеку, демо или чужой код; сэкономить время на рутинной настройке.
   · Боли: Потеря времени на чтение устаревшей документации, конфликты зависимостей в основной системе.
3. Преподаватели и менеджеры проектов
   · Потребности: Быстро развернуть демо-версии для студентов или коллег; обеспечить единую среду для всех.
   · Боли: Трата времени на поддержку студентов/коллег с проблемами окружения.
4. Тестировщики и QA-инженеры
   · Потребности: Быстрое развертывание билдов для тестирования в изолированном окружении.
   · Боли: Зависимость от DevOps-команд для настройки окружения.

2.2. Сценарии использования (User Stories)

· Как начинающий разработчик, я хочу запустить пример React-приложения с GitHub, чтобы изучить его работу, не разбираясь в настройке Webpack и Node.js.
· Как опытный инженер, я хочу быстро протестировать новую CLI-утилиту на Rust, не устанавливая компилятор и зависимости в свою основную систему.
· Как преподаватель, я хочу дать студентам одну ссылку на приложение, которое автоматически развернет лабораторную работу со всем необходимым ПО.
· Как security-исследователь, я хочу безопасно запустить потенциально подозрительный скрипт в изолированной среде, чтобы проанализировать его поведение.

3. Основные функции и функциональность

3.1. Ядро приложения (Core Flow)

1. Ввод репозитория:
   · Поддержка вставки URL GitHub (HTTPS, SSH) или прямого ввода owner/repo.
· Дополнительно: Перетаскивание .zip архива репозитория, выбор из истории, поиск через интеграцию с GitHub API.
   · Критерии приемки: Приложение корректно обрабатывает валидные URL GitHub и извлекает из них владельца и имя репозитория.
2. Анализ проекта (Project Detection & Analysis):
   · Сканирование структуры файлов репозитория для определения типа проекта (Python, Node.js, Rust, Java, Docker и т.д.).
   · Анализ конфигурационных файлов: package.json, requirements.txt, Cargo.toml, Dockerfile, docker-compose.yml, pyproject.toml, go.mod и других.
   · Определение рекомендуемой команды запуска на основе:
     · Стандартных полей (scripts.start в package.json).
     · Популярных шаблонов (например, main.py, index.js).
     · Анализа README.md (с помощью LLM для сложных случаев).
   · Критерии приемки: Для 10 тестовых репозиториев разных типов приложение корректно определяет стек технологий и предлагает логичную команду запуска.
3. Управление окружением (Environment Orchestration):
   · Автоматическое создание изолированного окружения для проекта.
   · Опциональная песочница (Sandbox Mode): Запуск в Docker-контейнере (рекомендуется по умолчанию для неизвестных репозиториев).
   · Прямой режим (Direct Mode): Запуск в виртуальном окружении языка (venv, virtualenv) или системном окружении (по выбору опытного пользователя).
   · Установка системных зависимостей (через пакетный менеджер ОС внутри контейнера или виртуальной среды).
   · Установка зависимостей языка (npm install, pip install, cargo build и т.д.).
   · Критерии приемка: Проект запускается в изолированном окружении без конфликтов с системными пакетами пользователя.
4. Запуск и отображение (Execution & Display):
   · Выполнение определенной команды запуска.
   · Для веб-приложений: Автоматическое открытие браузера на нужном localhost:port (порт определяется из логов или конфигурации).
   · Для GUI-приложений: Отображение графического интерфейса в отдельном окне (через проброс X11/Wayland для Linux, интеграцию с Windows/macOS GUI).
   · Для CLI-приложений: Отображение интерактивного терминала с выводом внутри приложения AutoLaunch.
   · Критерии приемки: Пользователь видит запущенное приложение (в браузере, окне или терминале) без ручных действий.
5. Управление жизненным циклом (Lifecycle Management):
   · Кнопки "Остановить", "Перезапустить", "Приостановить" для запущенного проекта.
   · Автоматическая очистка временных ресурсов (контейнеров, виртуальных сред) после остановки (опционально).
   · Просмотр логов выполнения (установки зависимостей, сборки, запуска) в красивом и структурированном виде внутри приложения.
   · Критерии приемка: Пользователь может контролировать состояние запущенного проекта и очищать ресурсы.

3.2. Функции безопасности (Safety Features)

1. Предупреждения об опасных командах:
   · Статический анализ скриптов и команд на наличие потенциально опасных операций (например, rm -rf /, curl | bash, запросы sudo).
   · Четкие предупреждения перед выполнением с объяснением риска.
   · Возможность отменить запуск или продолжить на свой страх и риск.
   · Критерии приемки: При анализе тестового репозитория с командой rm -rf /tmp/test выдается понятное предупреждение.
2. Изоляция через песочницу (Sandbox Mode):
   · Реализация на основе Docker с ограниченными правами (без root, read-only файловые системы, ограничение сетевого доступа).
   · Для пользователей без Docker — fallback на более легковесные механизмы изоляции (например, bubblewrap для Linux, sandbox-exec для macOS).
   · Критерии приемки: Зловредный тестовый скрипт, пытающийся записать файл в домашнюю директорию хоста, не может этого сделать в Sandbox Mode.
3. Система доверия (Trust System):
   · Возможность пометить репозиторий/автора как "доверенный" для последующих запусков без строгой изоляции.
   · Визуальные индикаторы уровня безопасности текущего запуска.

3.3. Удобства и дополнительные функции (Convenience Features)
1. Сохранение проекта (Project Snapshot):
   · Сохранение полного состояния настроенного и запущенного проекта (включая зависимости, виртуальное окружение) в локальную директорию.
   · Возможность позднее открыть этот "снимок" и запустить его мгновенно, без повторной настройки.
   · Критерии приемки: Пользователь сохраняет проект, закрывает AutoLaunch, после открытия сохраненного проекта он запускается за <10 секунд.
2. Менеджер проектов (Project Manager):
   · История запущенных репозиториев с возможностью быстрого перезапуска.
   · Теги, поиск и организация проектов по категориям.
   · Критерии приемка: Пользователь может найти проект, запущенный 2 недели назад, по имени репозитория или тегу.
3. Настройки и кастомизация:
   · Настройка предпочтительного режима изоляции (песочница/прямой).
   · Настройка путей для сохранения снимков, кэша зависимостей.
   · Выбор темы оформления (темная/светлая).

4. Рекомендации по техническому стеку

4.1. Фронтенд (GUI приложение)

· Рекомендуемая технология: Tauri (Rust + Web Technologies)
  · Обоснование: На основании исследования, Tauri предоставляет наилучший баланс для данного продукта:
    · Размер и производительность: Генерирует очень маленькие исполняемые файлы (~10 МБ против ~100 МБ у Electron) и использует меньше памяти, так как использует системный WebView.
    · Безопасность: Rust как язык бэкенда обеспечивает memory safety и снижает уязвимости. Tauri имеет строгую модель безопасности для IPC.
    · Кроссплатформенность: Поддержка Windows, macOS, Linux из одной кодовой базы.
    · Интеграция с системой: Отличные возможности для нативных вызовов (работа с Docker, файловой системой, процессами) через Rust.
  · Альтернативы:
    · Electron (TypeScript/JavaScript): Больше ресурсов, но огромная экосистема и простота разработки. Не рекомендуется из-за требований к производительности и размеру.
    · Flutter Desktop (Dart): Хорошая производительность и UI, но менее зрелая экосистема для десктопа и сложная интеграция с низкоуровневыми системными вызовами.
    · Нативные фреймворки (Qt, .NET MAUI): Максимальная производительность и интеграция, но требуется поддержка нескольких кодовых баз.

4.2. Бэкенд (Логика приложения) и анализ

· Ядро на Rust: В рамках Tauri, основная логика (анализ файлов, управление процессами, работа с Docker API) пишется на Rust.
  · Преимущества: Высокая производительность, безопасность, отсутствие runtime, отлично подходит для системного программирования.
· Анализ и эвристика:
  · Детерминированные правила: Набор правил (например, if file exists("package.json") -> Node.js project) для быстрого определения типа проекта.
  · Интеграция с LLM (опционально, для MVP v2): Использование локальной (например, Ollama с моделью Codellama) или облачной LLM API для анализа сложных README и определения команд запуска. Требует осторожности из-за затрат и задержек.

4.3. Технологии изоляции и управления окружением

· Основная песочница: Docker/Podman.
  · Использование Docker Engine API для создания и управления контейнерами.
  · Автогенерация Dockerfile для проектов без него, на основе обнаруженного стека.
· Fallback-изоляция:
  · Linux: bubblewrap (как в Flatpak), firejail.
  · macOS: sandbox-exec.
  · Windows: Windows Sandbox (ограниченно), AppContainer (сложно).
· Виртуальные окружения языков:
  · Python: venv/virtualenv.
  · Node.js: Локальная установка в node_modules.
  · Rust/Go: Локальная сборка в целевой директории.

4.4. Сторонние зависимости и API

· GitHub REST API / GraphQL API: Для получения информации о репозитории (по умолчанной ветке, структуре через Git Trees), поиска.
· Docker Engine API: Для управления жизненным циклом контейнеров.
· (Опционально) LLM API (OpenAI, Anthropic, локальная Ollama): Для "умного" анализа README.

5. Концептуальная модель данных

5.1. Основные сущности
1. Project (Проект)
   · id: UUID (уникальный идентификатор).
   · github_url: STRING (исходный URL).
   · owner: STRING (владелец репозитория).
   · repo_name: STRING (имя репозитория).
   · local_path: STRING (путь к локальной копии/снимку).
   · detected_stack: ENUM (Python, Nodejs, Rust, Docker, Unknown).
   · trust_level: ENUM (Unknown, Trusted, Untrusted).
   · created_at: TIMESTAMP.
2. ProjectSnapshot (Снимок проекта)
   · id: UUID.
   · project_id: UUID (FK to Project).
   · snapshot_path: STRING (путь к сохраненной директории/образу).
   · environment_type: ENUM (DockerImage, Venv, Raw).
   · metadata: JSON (сохраненные команды, порты, переменные окружения).
   · created_at: TIMESTAMP.
3. Execution (Запуск)
   · id: UUID.
   · project_id: UUID.
   · status: ENUM (Preparing, Running, Stopped, Failed).
   · sandbox_mode: BOOLEAN.
   · container_id: STRING (если Docker).
   · pid: INTEGER (если прямой запуск).
   · logs: TEXT.
   · started_at: TIMESTAMP.
   · finished_at: TIMESTAMP.

5.2. Хранение данных

· Локальная база данных: SQLite (встроенная в Tauri через sqlx или rusqlite).
· Хранение файлов:
  · Кэш зависимостей: ~/.autolaunch/cache/
  · Снимки проектов: ~/Documents/AutoLaunch/Projects/ (настраиваемо)
  · Конфигурация: ~/.autolaunch/config.toml

6. Принципы дизайна пользовательского интерфейса

6.1. Философия UX

· "Волшебство одной кнопки": Основной поток (вставить ссылку -> запустить) должен занимать не более 2-3 кликов.
· Прозрачность: Даже при автоматической работе, пользователь должен видеть, что происходит (этапы анализа, установки) через прогресс-бар и читаемые логи.
· Контроль: Опытные пользователи должны иметь доступ к расширенным настройкам и информации (полные логи, конфигурация контейнера).

6.2. Визуальный концепт (Wireframe в уме)

1. Главное окно:
   · Крупное текстовое поле с плейсхолдером "Вставьте ссылку на GitHub репозиторий..." в центре.
   · Кнопка "Запуск" или "Волшебная пуля" рядом.
   · Сверху: панель с историей последних проектов, настройками.
2. Окно процесса:
   · После запуска появляется модальное или панельное окно с:
     · Визуализированными шагами: "Анализ", "Настройка окружения", "Установка", "Запуск".
     · Красивым, сворачиваемым логом с подсветкой синтаксиса.
     · Кнопками управления (Стоп, Пауза, Сохранить снимок).
3. Менеджер проектов:
   · Боковая панель или отдельное окно с карточками проектов (иконка, название, дата последнего запуска, теги).
4. Настройки:
   · Чистый, сгруппированный интерфейс с переключателями (Режим песочницы по умолчанию, Автоочистка) и путями.

6.3. Дизайн-система

· Современный, минималистичный, "разработческий" стиль. Возможно, в духе VSCode или GitHub Desktop.
· Обязательная поддержка темной и светлой темы.
· Использование иконок для статусов и действий.

7. Соображения по безопасности

1. Изоляция как основа: По умолчанию ВСЕ неизвестные проекты запускаются в режиме песочницы (Docker с ограничениями).
2. Анализ кода: Статический анализ скриптов запуска и package.json/т.п. на предмет известных вредоносных паттернов.
3. Ограничение прав контейнера:
   · --read-only корневая ФС.
   · --cap-drop ALL (удаление всех привилегий).
   · --security-opt no-new-privileges.
   · Ограничение сетевого доступа только к локальному хосту (если не требуется иное).
4. Четкие предупреждения: Любая попытка выполнить команду с sudo, rm -rf вне временной директории, сетевые запросы наружу — требует явного подтверждения.
5. Хранение данных: Конфиденциальные данные пользователя (GH токены, если будут) должны храниться в зашифрованном виде (используя системные keyring).
6. Обновления: Механизм автоматических обновлений приложения для своевременного исправления уязвимостей.

8. Этапы/вехи разработки (Roadmap)

Фаза 1: Прототип ядра (Months 1-2)

· Цель: Доказательство концепции для одного стека.
· Результат: CLI-утилита на Rust, которая может:
  · Принять ссылку на простой Python (Flask) или Node.js (Express) проект.
  · Создать Dockerfile, собрать образ и запустить контейнер.
  · Пробросить порт и открыть браузер.
· Команда: 1-2 разработчика.
Фаза 2: Минимальный жизнеспособный продукт (MVP) (Months 3-5)

· Цель: Рабочее десктопное приложение для основных стеков.
· Результат: Приложение на Tauri с GUI, поддерживающее:
  · Python (Flask/Django), Node.js (React/Express), статические HTML сайты.
  · Основной поток: ввод URL -> запуск в Docker -> открытие в браузере.
  · Базовый интерфейс с логами и кнопкой "Стоп".
  · Система предупреждений об опасных командах.
· Команда: 2-3 разработчика (Rust/TAURI, фронтенд).

Фаза 3: Расширение и полировка (Months 6-9)

· Цель: Полноценный продукт для широкой аудитории.
· Результат:
  · Поддержка 10+ основных стеков (Rust, Go, Java, PHP, Ruby, Docker Compose).
  · Реализация режима песочницы и прямого режима.
  · Функция "Сохранить снимок".
  · Менеджер проектов с историей и тегами.
  · Улучшенный анализ (базовая эвристика + простой анализ README).
  · Настройки и кастомизация.
· Команда: 3-4 человека (добавляется QA/дизайнер).

Фаза 4: Продвинутые функции и рост (Months 10-12+)

· Цель: Лидерство в niche и платформенные возможности.
· Результат (гипотезы):
  · Интеграция LLM для "понимания" сложных инструкций.
  · Плагины/рецепты от сообщества для экзотичных проектов.
  · Облачные снимки (запуск проекта в облачном окружении по ссылке).
  · Магазин шаблонов быстрого запуска популярных инструментов (типа localhost:3000 для готовых демо).
  · Публичный API для интеграции в CI/CD и другие инструменты.

9. Потенциальные вызовы и решения

1. Вызов: Определение команды запуска в хаотичных проектах.
   · Решение: Многоуровневая стратегия: 1) Поиск стандартных конфигов → 2) Эвристика по популярным шаблонам → 3) Запрос к пользователю с предложением выбора из найденных возможных точек входа → 4) (В будущем) Анализ README через LLM.
2. Вызов: Запуск GUI-приложений в изолированном контейнере (особенно на Linux).
   · Решение: Использование -v /tmp/.X11-unix:/tmp/.X11-unix и -e DISPLAY для Linux. Для macOS и Windows использовать встроенные механизмы проброса GUI (например, VNC внутри контейнера или RDP). Для MVP можно ограничиться поддержкой веб и CLI.
3. Вызов: Производительность и размер приложения (особенно с Docker внутри).
   · Решение: Не включать Docker в дистрибутив. Требовать его предустановки. Сфокусироваться на оптимизации Tauri-приложения. Для fallback-изоляции использовать легковесные утилиты.
4. Вызов: Юридические вопросы и ответственность за запуск вредоносного кода.
   · Решение: Яркие предупреждения, лицензионное соглашение (EULA), снимающее ответственность, открытый код алгоритмов безопасности. Активная коммуникация о рисках.
5. Вызов: Поддержка огромного разнообразия стеков и инструментов сборки.
   · Решение: Начать с самых популярных (Top-5). Создать расширяемую архитектуру "детекторов" и "оракулов запуска", чтобы сообщество могло добавлять поддержку новых стеков через конфигурационные файлы или плагины.

10. Возможности для будущего расширения

1. Платформа для демо-проектов (Demo-as-a-Service): Авторы проектов могут прикреплять к репозиторию специальный файл .autolaunch, гарантирующий идеальный запуск через ваше приложение. Это может стать стандартом де-факто.
2. Образовательный инструмент: Режим "пошагового запуска", где приложение объясняет, какие шаги оно выполняет и почему. Интеграция с учебными платформами.
3. Корпоративная версия: Управление внутренними репозиториями, контроль политик безопасности, централизованное управление снимками окружений для команд разработки.
4. Интеграция с облачными провайдерами: Возможность запускать тяжелые проекты не на локальной машине, а в облачном инстансе с мощным GPU/CPU, с пробросом интерфейса к пользователю.
5. Магазин приложений: Сообщество может публиковать "однокликовые" версии сложных open-source инструментов (типа Metabase, GitLab) для локального запуска и тестирования.
